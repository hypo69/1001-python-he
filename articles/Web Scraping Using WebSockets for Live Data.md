### **סקרייפינג (Web Scraping)**
**למה כדאי להשתמש בווב-סוקט (WebSocket) כדי לקבל נתוני קריפטו בזמן אמת**

מאת: לאו ווי הונג

טיפ מציל חיים לקבלת נתונים המשתנים ללא הרף!

<br>

![תמונה מאת AbsolutVision ב-Unsplash](https://images.unsplash.com/photo-1621417534129-28185c4f279f?crop=entropy&cs=tinysrgb&fit=max&fm=jpg&ixid=MnwxOTc3M18w&ixlib=rb-1.2.1&q=80&w=1080)
<p style="text-align: center;">תמונה מאת AbsolutVision ב-Unsplash</p>

אחת הדרכים הטובות ביותר להפוך את תהליך איסוף הנתונים מאתר אינטרנט לאוטומטי היא לבנות זחלן רשת (web crawler) שיבצע את העבודה. ניתן ליצור זחלן רשת שמגרד נתונים במגוון שיטות. לדוגמה, אם תרצו לאסוף נתונים מאתר פעם בכמה שעות או פעם בשבוע, תוכלו לכתוב תוכנית שתשלח בקשת HTTP באופן אוטומטי או תפתח דפדפן כדי לעשות זאת, מכיוון שהתדירות נמוכה. עם זאת, אם אתם חושבים על גירוד נתונים תוך שניות, עדיין עדיף להתחבר דרך ווב-סוקט.

לפני שנעבור לחלק המעשי של כתיבת קוד לסקרייפר, בואו נעבור בקצרה על מה זה ובאילו מקרים בדרך כלל משתמשים בזה.

**בואו נכניס את זה להקשר מהחיים האמיתיים…**

יום אחד, אמרתם לחברה שלכם שתתקשרו אליה כדי לדון במושג שאינכם מבינים. אז, כשהתקשרתם לחברה (לחיצת יד), היא תבדוק מי מתקשר ואז תחליט אם לענות לשיחה (אימות). לאחר שהחברה שלכם השלימה את האימות וענתה לשיחה, היא תסביר את המושג ותשאיר את הקו פתוח לשאלות נוספות שלכם (הזרמת נתונים מהשרת).

<br>

![תרשים זרימה שמסביר איך נראה תהליך ווב-סוקט](https://miro.medium.com/v2/resize:fit:816/1*82o6cKx-b7k_1-x5M9zKSA.png)
<p style="text-align: center;">תרשים זרימה שמסביר איך נראה תהליך ווב-סוקט</p>

בדרך כלל, כדאי לשקול להשתמש בווב-סוקט כדי לאסוף נתונים במקרים כמו מטבעות קריפטוגרפיים, מחירי מניות בזמן אמת, הימורים חיים, הודעות מיידיות, פלטפורמות מסחר, ועוד רבים.

לאחר שהבנו את המושגים, בואו נדמיין תרחיש אמיתי ונעבוד משם. לדוגמה, אם תרצו לבנות בוט מסחר חי כדי לסחור במטבעות קריפטוגרפיים, וכבר יש לכם סט חוקים מוכן כדי לנצח את השוק.

**מה השלב הבא?**

תצטרכו למצוא אתר שממנו תוכלו לאסוף נתונים בזמן אמת. אתם נתקלים בכמה ממשקי API (API) אבל חוששים שהם ייכשלו, מה שעלול להוביל להפסד כספי. לכן, אתם מחליטים למצוא אתר שמספק מחירי קריפטו בזמן אמת. בואו נבחר ב-**Cryptocompare** (ספקית נתוני שוק קריפטו גלובלית) לצורך ההדגמה.

### **חקירה**

כאשר אתם נכנסים לאתר זה באמצעות דפדפן כרום, לחצו קליק ימני, בחרו "בדוק" (Inspect), ולאחר מכן לחצו על לשונית **WS**. אתם אמורים לראות תצוגה דומה לצילום המסך למעלה. תבחינו שכתובת ה-URL של הבקשה (request url) עבור הווב-סוקט הזה היא `wss://streamer.cryptocompare.com/v2?format=streamer`.

בשלב הבא, בואו נעבור ללשונית ההודעות (Messages) כדי לקבל את ההודעה הדרושה לשליחה לשרת על מנת להשלים את האימות.

<br>

![לשונית ההודעות](https://miro.medium.com/v2/resize:fit:1400/1*98tM6-JqD5G9n-22pD74Xw.png)

עכשיו, תראו שני סוגי חצים -

*   ההודעות שאחרי **החצים הירוקים** מציינות את ההודעות היוצאות שהדפדפן שלכם שולח לשרת.
*   ההודעות שאחרי **החצים האדומים** הן הנתונים הנכנסים שהדפדפן שלכם מקבל מהשרת.

עד כאן, אתם רק צריכים להעתיק את ההודעות של החצים הירוקים על ידי לחיצה ימנית על ההודעה ובחירה ב"העתק הודעה" (copy message). תצטרכו את 5 ההודעות הללו בחלק של הקוד.

### **כתיבת הקוד**

צירפתי קטע קוד כדי שתוכלו לנסות. משהו שתצטרכו לשים לב אליו הוא לשנות את מפתח ה-WebSocket (`Sec-WebSocket-Key`) למפתח שאתם יכולים למצוא בדפדפן שלכם תחת לשונית "Headers". דוגמה למטה:

<br>

![מציאת מפתח ה-Sec-WebSocket-Key שלכם](https://miro.medium.com/v2/resize:fit:1400/1*5T5W44dC2k5fXjFqj2iY6w.png)
<p style="text-align: center;">מציאת מפתח ה-Sec-WebSocket-Key שלכם</p>

הנה סיכום של מה שקטע הקוד למעלה עושה:

1.  יצירת חיבור עם כתובת ה-URL של הבקשה.
2.  שליחת 5 ההודעות היוצאות לשרת.
3.  לשבת ולהירגע בזמן שמחכים להודעות הנכנסות, שבדרך כלל מגיעות תוך שניות.

### **תוצאה**

שמרו את קטע הקוד לקובץ פייתון והריצו אותו. אתם אמורים לראות פלט דומה.

### **בעיות שאתם עלולים להיתקל בהן**

הנה כמה מהבעיות שאתם עלולים להיתקל בהן וכיצד ניתן לפתור אותן.

**1. הודעת שגיאה: Connection is already closed (החיבור כבר נסגר)**
אם אתם רואים הודעה זו, המשמעות היא בדרך כלל שהשרת מנסה למנוע מכם להתחבר. הדרך המהירה ביותר לפתור זאת היא להחליף את שרתי הפרוקסי או ה-VPN שלכם.

**2. קוד סטטוס HTTP: 401 (לא מורשה - Unauthorized)**
תגובת 401 פירושה שהשרת אינו מקבל את בקשת החיבור שלכם. אם השרת עובד כשורה, הבעיה היא בבקשת החיבור שלכם. בקיצור, בדרך כלל זה אומר שאתם משתמשים באישורים (credentials) שגויים. כדי לפתור זאת, בדקו את כותרות הבקשה (request headers) ושנו אותן בהתאם.

**3. קוד סטטוס HTTP: 403 (אסור - Forbidden)**
בדומה ל-401, אך במקרה זה, אימות מחדש לא יעזור. ייתכן שהמשמעות היא שהלוגיקה המקורית של היישום השתנתה, למשל שתהליך לחיצת היד השתנה. לכן, כדי לוודא שהכל רץ חלק, בדקו את כותרות הבקשה שלכם ואת ההודעות שאתם צריכים לשלוח לשרת.

### **ווב-סוקטים בקריפטו: הזרמת נתונים בזמן אמת**

מאת: Oreoluwa F. | ידע | 17 באוקטובר 2024 | (0)

האם אתם חובבי קריפטו, אולי סוחרים או מפתחים, המעוניינים להקדים את השוק? אתם יודעים שכל שנייה קובעת כשמדובר בקריפטו, ועליכם להיות מסוגלים לגשת לנתונים בזמן אמת אם אתם רוצים להימנע מהחמצת הזדמנויות ולהצליח במסחר. אולי עבדתם עם ממשקי API בעבר ומצאתם אותם שימושיים. עם זאת, כעת אתם צריכים משהו מהיר יותר, יעיל יותר ומגיב יותר. כאן נכנסים לתמונה הווב-סוקטים בעולם הקריפטו.

ווב-סוקטים בקריפטו הם פרוטוקול תקשורת השומר על חיבור רשת ומאפשר זרימה חלקה של תקשורת דו-כיוונית בזמן אמת בין הדפדפן שלכם, שהוא הלקוח, לבין השרת. בעוד שממשקי API מסורתיים פועלים על בסיס בקשה-תגובה, ווב-סוקטים מבטיחים שקיים חיבור קבוע בין המערכת שלכם לשרת. זה מאפשר לדפדפן שלכם לקבל עדכונים מיידיים מהשרת מבלי צורך לשלוח בקשות חדשות בכל פעם שיש צורך במידע חדש. ווב-סוקטים הם משני-משחק בתחום הקריפטו, שבו הזמן הוא תמיד גורם מכריע.

### **מגבלות של ממשקי API מסורתיים**

ייתכן שאתם כבר מכירים ממשקי API מסוג RESTful, המשמשים בדרך כלל לאינטראקציה עם בורסות קריפטו. ממשקי API אלה פועלים על מנגנון תשאול (polling). הלקוח שולח בקשת HTTP לשרת, והשרת מגיב עם הנתונים העדכניים ביותר. מודל זה עשוי להתאים ליישומים רבים הדורשים עדכוני נתונים, אך הוא אינו מספק בתחום הקריפטו, שבו עדכון מתמיד של הנתונים האחרונים הוא בעל חשיבות עליונה בקבלת החלטות.

לדוגמה, דמיינו שאתם עוקבים אחר מחיר הלייטקוין (Litecoin) במגוון בורסות או פלטפורמות מסחר. עם API מסוג RESTful, תצטרכו לשלוח בקשות לכל בורסה באופן קבוע כדי לבדוק עדכוני מחירים אחרונים. שיטה זו צורכת רוחב פס ומגדילה את ההשהיה (latency). עד שתקבלו תגובה מהבורסה, המחיר כבר עלול להשתנות, ולהשאיר אתכם עם מידע לא עדכני. גרוע מכך, ככל שאתם מבצעים תשאול בתדירות גבוהה יותר, כך אתם מפעילים יותר עומס על השרת ועל המכשיר שלכם.

כאן ווב-סוקטים בקריפטו מנצחים. הם מציעים פתרון על ידי שמירה על חיבור קבוע ופתוח, מה שמאפשר לכם לקבל תמיד עדכונים מבורסות מרובות בזמן אמת, מבלי צורך לשלוח בקשות בכל פעם. ברגע שנוצר חיבור בין הלקוח לשרת, השרת דוחף אליכם מידע חדש באופן אוטומטי, ומאפשר לכם לקבל החלטות מושכלות יותר על סמך נתונים מדויקים.

### **איך ווב-סוקטים עובדים?**

כעת בואו נבחן כיצד ווב-סוקטים משיגים את המטרה של יצירת חיבור פתוח וקבוע בין הלקוח לשרת לצורך הזרמת נתונים רציפה. התהליך מתחיל כאשר הלקוח שולח בקשה ראשונית לשרת. הבקשה משתמשת ב-HTTP, שהוא תקן סטנדרטי לביצוע קריאות API. עם זאת, לצד הבקשה ישנה כותרת שדרוג (upgrade header) מיוחדת המבקשת מהשרת להחליף את החיבור לפרוטוקול ווב-סוקט. ברגע שהשרת מסכים לבקשה זו, החיבור משודרג מ-HTTP לווב-סוקט. חיבור זה נשאר פתוח, ומאפשר תקשורת דו-כיוונית רציפה בין הלקוח לשרת. משמעות הדבר היא ששני הצדדים יכולים כעת לשלוח ולקבל הודעות בזמן אמת מבלי לסגור את החיבור לאחר כל הודעה.

לאחר שנוצר, החיבור מאפשר לשרת להתחיל לדחוף נתונים ללקוח ברגע שיש עדכון חדש. הלקוח יכול לקבל התראות על מגוון רחב של אירועים, כולל שינויי מחירים, פקודות חדשות, ביצועי עסקאות ועוד הרבה יותר. מכיוון שתקשורת דו-כיוונית אפשרית עם ווב-סוקטים, הלקוח יכול גם לשלוח הודעות לשרת בעת הצורך, מה שהופך את האינטראקציה לאינטראקציה דו-כיוונית אמיתית. כאשר אינכם זקוקים יותר לנתונים, או שאתם רוצים להתנתק, תוכלו לסגור את החיבור בקלות. פעולה זו מסיימת את חיבור הווב-סוקט בצורה נקייה. כתוצאה מכך, משאבים משתחררים בשני קצות החיבור לשימוש במצב אחר.

### **ווב-סוקטים ומדרגיות (Scalability)**

ווב-סוקטים מציעים מדרגיות פשוטה יותר בקריפטו. שקלו, למשל, שאתם מנהלים אתר או יישום קריפטו עם תעבורה גבוהה, שבו אלפי משתמשים עוקבים כל הזמן אחר תנועות מחירים בו-זמנית. שימוש ב-API מסוג RESTful ידרוש מכם לעבד אלפי בקשות בדקה. זה מפעיל עומס עצום על השרתים שלכם. עם זאת, עם ווב-סוקטים, כל מה שאתם צריכים הוא חיבור יחיד ונרחב לכל משתמש. זה מפחית באופן דרמטי את התקורה והמורכבות הכרוכים בניהול עדכונים בזמן אמת עבור בסיסי משתמשים גדולים.

יתר על כן, ווב-סוקטים מוסיפים שכבת גמישות נוספת על ידי טיפול במנויים של משתמשים לערוצים אחרים, כגון ביצועי עסקאות, סיכומי שוק והתראות חדשותיות, בנוסף לעדכוני המחירים העיקריים שהם מספקים. גמישות זו מאפשרת לכם להרחיב את היישום שלכם בקלות מבלי לדאוג לעומס יתר על השרת.

### **מגבלות של ווב-סוקטים בקריפטו**

עד כה, דנו בכמה מהיתרונות של ווב-סוקטים בקריפטו. עם זאת, לווב-סוקטים יש כמה מגבלות ביישומם. בואו נבחן כמה מהן.

**1. יציבות החיבור**
אחד האתגרים בשימוש בווב-סוקטים בקריפטו הוא ניהול יציבות החיבור. מכיוון שהחיבור הוא קבוע, כל שיבוש ברשת, בין אם מהלקוח או מהשרת, יכול להפריע לזרימת הנתונים. בשווקים תנודתיים, אפילו השבתות קצרות יכולות להוביל לעדכונים שהוחמצו או לפעולות מושהות, מה שעלול להיות בעל השלכות חמורות על מסחר או ניהול תיקי השקעות. עליכם ליישם מנגנוני התחברות מחדש ולנטר באופן רציף את סטטוס החיבור כדי להבטיח שהמערכת שלכם תישאר מסונכרנת עם השרת. זה מוסיף שכבת מורכבות נוספת לתהליך הפיתוח שלכם, שכן שמירה על חיבור יציב חיונית להזרמת נתונים בזמן אמת.

**2. חששות אבטחה**
ווב-סוקטים בקריפטו הם חזקים, אך הם גם פותחים פגיעויות אבטחה פוטנציאליות. בגלל הזרמת הנתונים הרציפה, חיוני ליישם אמצעי הצפנה ואימות חזקים כדי להגן על החיבור. ללא פרוטוקולי אבטחה ראויים כמו WSS (WebSocket Secure), אתם מסתכנים בחשיפת מידע רגיש, כגון אישורי משתמש או אסטרטגיות מסחר, לגורמים זדוניים. בתחום הקריפטו, שבו האבטחה היא מעל הכל, הבטחת אבטחת העברת הנתונים היא קריטית. תצטרכו גם להיות מודעים להתקפות כגון התקפות מניעת שירות מבוזרת (DDoS) שיכולות להתמקד בחיבורי ווב-סוקט קבועים ולגרום להשבתות או לדליפות נתונים.

**3. תקורה עבור משימות פשוטות**
בעוד שווב-סוקטים בקריפטו הם אידיאליים להזרמת נתונים בתדירות גבוהה ובזמן אמת, הם עשויים להיות פתרון מוגזם למשימות פשוטות יותר. לא כל אינטראקציה עם שרת דורשת חיבור קבוע, ובמקרים מסוימים, שימוש בווב-סוקטים עלול להוסיף מורכבות מיותרת. אם אתם צריכים רק עדכונים מזדמנים או מבצעים אחזור נתונים בנפח נמוך, ממשקי API מסורתיים מבוססי HTTP יכולים להיות מעשיים יותר. ממשקי API אלה מספקים גישה פשוטה יותר, שכן הם אינם דורשים שמירה על חיבור פתוח, מה שהופך אותם לקלים יותר לניהול עבור פעולות בעלות סיכון נמוך או פחות רגישות לזמן. שימוש בווב-סוקטים בקריפטו למשימות כאלה עלול להוביל לצריכת משאבים גבוהה יותר ולמאמץ פיתוח גדול מהנדרש.

**4. מורכבות בצד השרת**
יישום ווב-סוקטים בקריפטו אינו אתגר רק בצד הלקוח; השרת דורש גם ניהול קפדני. תמיכה במספר גדול של חיבורי ווב-סוקט בו-זמניים דורשת משאבי שרת משמעותיים. לדוגמה, שמירה על אלפי חיבורים פתוחים בו-זמנית עלולה להעמיס על זיכרון השרת וה-CPU אם לא מנוהלת כראוי. זה יכול לגרום לצווארי בקבוק, זמני תגובה איטיים יותר, או אפילו קריסות אם התשתית אינה מתוכננת להתמודד עם העומס. בנוסף, מנגנוני איזון עומסים (load balancing) וגיבוי (failover) הופכים מורכבים יותר עם ווב-סוקטים מכיוון שיש לשמור על מצב החיבור בין שרתים. עליכם לוודא שתצורת השרת שלכם מסוגלת להתמודד עם הדרישות המוגברות של חיבורים קבועים, מה שעשוי לדרוש השקעה רבה יותר הן בתשתיות והן במומחיות.

**ש: מהו ווב-סוקט בבלוקצ'יין?**
**ת:** ווב-סוקטים בבלוקצ'יין מציעים דרך דינמית לגשת לנתונים בזמן אמת, ומחוללים מהפכה בדרך שבה מפתחים ומשתמשים מתקשרים עם רשתות מבוזרות. על ידי יצירת חיבור קבוע בין הלקוח לשרת, ווב-סוקטים מאפשרים העברת נתונים מיידית, המאפשרת לכם לקבל עדכונים על עסקאות, אישורי בלוקים ואירועי חוזים חכמים ברגע שהם קורים. גישה מיידית זו חיונית בעולם המהיר של מטבעות קריפטוגרפיים, שבו תזמון יכול להשפיע באופן משמעותי על החלטות מסחר וחווית משתמש. עם ווב-סוקטים, תוכלו ליצור יישומים המגיבים באופן מיידי לפעילות בבלוקצ'יין, מה שמקל על מעקב אחר שינויים ומעורבות בנוף המתפתח של נכסים דיגיטליים.

**ש: האם ווב-סוקט תומך בהזרמה (Streaming)?**
**ת:** ווב-סוקטים בהחלט תומכים בהזרמה, וזה מה שהופך אותם לחזקים כל כך ביישומים בזמן אמת כמו קריפטו. כאשר אתם משתמשים בווב-סוקטים, אתם יוצרים חיבור רציף בין לקוח לשרת, המאפשר לנתונים לזרום באופן מיידי ודו-כיווני. בניגוד לבקשות HTTP מסורתיות, שבהן עליכם לבצע קריאות מרובות לעדכונים, ווב-סוקטים מזרים נתונים ברגע שהם זמינים. חיבור חי זה הוא מה שמניע את זרימת הנתונים המהירה והבלתי פוסקת שאתם רואים בדברים כמו עדכוני מחירים בקריפטו, שינויים בספר פקודות והתראות שוק - ושומר על הכל מגיב ועדכני עד לשנייה האחרונה.

**ש: מהי המגבלה של ווב-סוקטים ב-Binance?**
**ת:** כשמדובר בשימוש בווב-סוקטים עם Binance, יש כמה מגבלות חשובות שיש לזכור. כל חיבור בודד יכול להאזין למקסימום של 1,024 ערוצים (streams), מה שמציע גמישות רבה למעקב אחר עדכוני שוק וצמדי מסחר שונים. עם זאת, Binance גם מטילה מגבלה של 300 חיבורים לכל כתובת IP מדי חמש דקות. משמעות הדבר היא שאם אתם צריכים לנטר נכסים או ערוצי נתונים מרובים, תצטרכו לתכנן את החיבורים שלכם בחוכמה כדי להישאר בגבולות אלה. הבנת מגבלות אלה מסייעת לכם לייעל את אסטרטגיית המסחר שלכם ומבטיחה שתפיקו את המרב מיכולות הנתונים בזמן אמת של Binance.
