
### 🧠 מהו ההבדל בין `git merge` ל־`git rebase`, ומתי כדאי להשתמש בכל אחד מהם?

A. `merge` משכתב את ההיסטוריה ויוצר גרסה חדשה נקייה, בעוד `rebase` משלב שינויים ללא יצירת commit נוסף.
B. `merge` מאחד שינויים משני ענפים תוך שמירה על כל היסטוריית הפיצול, בעוד `rebase` מעביר את הבסיס של הענף למיקום חדש, כך שההיסטוריה תהיה ליניארית.
C. `merge` דורש מחיקה של ענפים לאחר השילוב, בעוד `rebase` מחייב פתרון קונפליקטים בכל מקרה.
D. אין הבדל – שניהם תמיד יוצרים commit חדש ומומלצים לשימוש זהה.

**✅ תשובה נכונה: B**

---

### 🔍 הסבר:

ב־Git קיימות שתי שיטות עיקריות לשילוב שינויים מענף אחד לשני:

#### `git merge`:

* משלב את ההיסטוריה של שני ענפים.
* שומר על מבנה "ענפי" – ניתן לראות מתי בוצע הפיצול והמיזוג.
* יוצר commit מיזוג חדש (`merge commit`) אם יש שינויים בשני הענפים.
* לא משכתב היסטוריה – בטוח לשימוש גם בשיתוף עם אחרים.

#### `git rebase`:

* מעביר את הבסיס (base) של ענף אחד לראש של ענף אחר.
* משכתב את ההיסטוריה כדי להפוך אותה לליניארית – אין פיצולים.
* אינו יוצר merge commit, אלא "מניח" את הקומיטים זה אחרי זה מחדש.
* מתאים בעיקר לעבודה מקומית ופרטית, לפני שליחה ל־remote (כגון GitHub).

---

### 📌 דוגמה:

נניח שיש לך שני ענפים: `main` ו־`feature`.

#### בעזרת merge:

```bash
git checkout feature
git merge main
# ייווצר merge commit אם יש שינויים בשני הצדדים
```

#### בעזרת rebase:

```bash
git checkout feature
git rebase main
# הקומיטים של feature "יוזזו" לראש main
```

### 💡 טיפים מעשיים:

1. **בעת שיתוף פעולה – העדף merge:**
   היסטוריה משותפת נשמרת, פחות בעיות בקונפליקטים.

2. **לפני פתיחת Pull Request – העדף rebase:**
   קוד נקי וליניארי – יתרון כשעובדים לבד או רוצים "להבריק" את ההיסטוריה.

3. **לא לבצע `rebase` על ענף שכבר שותף (pushed):**
   זה יגרום להיסטוריה שונה מה־remote ועלול לבלגן את העבודה של אחרים.

4. **קונפליקטים? אל תילחץ!**
   גם `merge` וגם `rebase` עשויים לדרוש פתרון קונפליקטים. אחרי פתרון, השתמש ב־`git rebase --continue` או `git commit`.

5. **השתמש ב־`git log --oneline --graph` כדי להבין מה קרה בהיסטוריה.**
