**שאלה 668.** מהי ירושה מרובה בפייתון, ובאילו מקרים השימוש בה מתאים? אילו בעיות עלולות להתעורר בעת שימוש בירושה מרובה, וכיצד ניתן להימנע מהן?

-   A. ירושה מרובה היא היכולת של מחלקה לרשת רק ממחלקת אב אחת, וזה מפשט את הארכיטקטורה.
-   B. ירושה מרובה היא כאשר מחלקה יורשת שיטות רק ממחלקת אב אחת.
-   C. ירושה מרובה היא כאשר מחלקה יורשת ממספר מחלקות בסיס ויכולה להשתמש בשיטות ובמאפיינים שלהן, אך זה עלול לגרום לבעיות, כולל "בעיית היהלום", שניתן להימנע ממנה באמצעות קומפוזיציה.
-   D. ירושה מרובה היא היכולת ליצור מחלקות עם פרמטרים שניתנים לשינוי בזמן ריצה.

**התשובה הנכונה: C**

**הסבר:**

ירושה מרובה (multiple inheritance) בפייתון היא היכולת של מחלקה לרשת מאפיינים ושיטות ממספר מחלקות אב (בסיס). זה מאפשר ליצור מחלקות המשלבות פונקציונליות ממקורות שונים, אך עלול לגרום לבעיות אם למספר מחלקות אב יש שיטות ומאפיינים זהים.

*   **עקרונות יסוד של ירושה מרובה:**
    *   **ירושה ממספר מחלקות:** מחלקת בן יכולה לרשת ממספר מחלקות אב (סופר-קלאסים).
    *   **שילוב פונקציונליות:** מאפשר לשלב ולהשתמש מחדש בקוד ממקורות שונים.
    *   **תמיכה מוגבלת:** ירושה מרובה מסבכת את היררכיית המחלקות, ויש להשתמש בה בזהירות.
*   **סדר ירושה:** בירושה מרובה, פייתון משתמשת ב-MRO (Method Resolution Order) כדי לקבוע כיצד נקראות ומעובדות שיטות.

*   **בעיית "ירושת היהלום" (Diamond problem):**
    *   מתעוררת כאשר מחלקת אב משותפת מעבירה את אותה שיטה או מאפיין למספר מחלקות בנות, ולאחר מכן מחלקת בן יורשת מהן.
    *   זה עלול להוביל לעמימות בקריאה לשיטות.
    *   פייתון משתמש באלגוריתם C3, שעוזר להימנע מעמימות על ידי הגדרת ה-MRO.

*   **מתי להשתמש בירושה מרובה:**
    *   כאשר מחלקה צריכה להכיל מספר קבוצות נפרדות של שיטות.
    *   כאשר יש צורך להוסיף פונקציונליות למחלקה מבלי לשנות את המבנה שלה.
*   **חלופה:**
    *   קומפוזיציה (composition) - זה כאשר אתה יוצר מחלקה הכוללת מחלקות אחרות כמאפיינים שלה. במקרים שבהם ניתן להימנע מירושה מרובה, קומפוזיציה היא בדרך כלל גישה עדיפה יותר.

**דוגמאות:**

```python
# דוגמה לירושה מרובה
class A:
    def method_a(self):
        print("Method A")

class B:
    def method_b(self):
        print("Method B")

class C(A, B): # ירושה מרובה
    def method_c(self):
        print("Method C")

obj_c = C()
obj_c.method_a()  # ידפיס: Method A
obj_c.method_b()  # ידפיס: Method B
obj_c.method_c()  # ידפיס: Method C


# דוגמה עם בעיית היהלום

class A2:
    def method(self):
        print("Method from class A")
class B2(A2):
    def method(self):
        print("Method from class B")
class C2(A2):
    pass
class D2(B2, C2):
    pass
obj_d = D2()
obj_d.method() # ידפיס: Method from class B (סדר הירושה משפיע על התוצאה)
print(D2.__mro__) # מציג את סדר רזולוציית השיטות (Method Resolution Order - MRO)

# דוגמה לקומפוזיציה
class Engine:
    def start(self):
        print("Engine started")

class Radio:
  def play(self):
    print("Radio is on")

class Car:
    def __init__(self):
        self.engine = Engine()
        self.radio = Radio()

    def drive(self):
       self.engine.start() # המנוע בשימוש
       self.radio.play() # וגם הרדיו.
car = Car()
car.drive()

```

*   **B. ירושה מרובה היא כאשר מחלקה יורשת שיטות רק ממחלקת אב אחת.:** שגוי.
*   **C. ירושה מרובה היא כאשר מחלקה יורשת ממספר מחלקות בסיס ויכולה להשתמש בשיטות ובמאפיינים שלהן, אך זה עלול לגרום לבעיות, כולל "בעיית היהלום", שניתן להימנע ממנה באמצעות קומפוזיציה.:** נכון.
